Optimiser une application .NET ne consiste pas à appliquer une liste de micro-optimisations. Une démarche sérieuse commence par la mesure, puis par la compréhension des coûts dominants.

## Mesurer avant de changer

Les premiers réflexes doivent être:

- profiler CPU
- observer allocations mémoire
- mesurer latence et throughput
- vérifier les accès I/O
- identifier les appels réseau ou base de données dominants

Sans ces mesures, on optimise souvent la mauvaise chose.

## Les sources classiques de lenteur

Dans les applications .NET, les problèmes fréquents sont:

- allocations inutiles
- requêtes SQL trop nombreuses
- sérialisations répétitives
- contention sur ressources partagées
- appels synchrones dans des flux asynchrones
- traitement réseau trop verbeux

Les problèmes d'architecture comptent souvent plus que les détails syntaxiques.

## La bonne approche

Un plan d'optimisation efficace suit souvent cet ordre:

1. mesurer
2. identifier le goulet dominant
3. corriger à la source
4. remesurer

Cette discipline evite les optimisations cosmétiques.

## Conclusion

La performance en .NET est un sujet d'observabilité et d'architecture avant d'être un sujet de tricks. Ce qui compte, c'est de réduire les coûts dominants là où ils pénalisent vraiment le service.
