La configuration initiale d'un projet .NET influence fortement sa capacité à rester lisible quand l'équipe, les fonctionnalités et les intégrations augmentent. Pourtant, beaucoup de projets sont démarrés avec une bonne intention mais sans conventions assez claires.

Configurer un projet .NET pour le long terme consiste surtout à préparer:

- une structure de solution compréhensible
- des dépendances claires
- une configuration multi-environnements propre
- une stratégie de tests
- des conventions de logging et d'observabilité

## 1. Fixer des frontières de projet

Une structure simple fonctionne déjà très bien:

```text
src/
  Project.Api/
  Project.Application/
  Project.Domain/
  Project.Infrastructure/
tests/
  Project.UnitTests/
  Project.IntegrationTests/
```

Cette séparation aide à éviter que tout remonte dans l'API.

## 2. Standardiser la configuration

Un projet .NET vieillit mal quand la configuration devient diffuse. Il faut centraliser:

- `appsettings.json`
- `appsettings.Development.json`
- variables d'environnement
- secrets locaux

La règle doit être simple: le code lit des options typées, pas des chaînes magiques partout.

## 3. Poser une stratégie de dépendances

Avant même d'écrire les fonctionnalités, il faut clarifier:

- où vivent les contrats
- où vivent les clients externes
- comment se fait l'injection de dépendances
- comment on introduit une nouvelle intégration

Cette clarté evite les dépendances transverses difficiles à corriger plus tard.

## 4. Activer les qualités de base

Un projet qui doit durer devrait démarrer avec:

- linting
- type safety
- tests
- conventions de logs
- gestion des erreurs centralisée

Ce n'est pas du luxe. C'est un minimum pour maintenir un rythme soutenable.

## Conclusion

Configurer un projet .NET pour le long terme, c'est réduire très tôt les zones d'ambiguïté. Un bon démarrage ne garantit pas une bonne architecture, mais un mauvais démarrage fabrique presque toujours de la dette.
